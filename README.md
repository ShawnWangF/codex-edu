# HKEIU 排程系统需求与方案（基于现有真实工作流）

## 1. 业务理解

HKEIU 需要完整映射以下核心对象与规则：

### 1.1 团组 vs 批次

- **团组**：由若干学校组成的执行单元，拥有自己的航班、人数、类型（小学/中学/自定义）与每日行程。
- **批次**：多个团组被打包到同一“到达/离开窗口”，通常共享同一班次航班窗口与接驳车辆逻辑。
- 关键约束：
  - 同批次团组在抵离日期上具有绑定关系。
  - 团组是行程编排与资源计算最小颗粒；批次是运输与总览管理颗粒。

### 1.2 色块日历的业务语义

在总览排程中，色块不只是视觉标记，而是**可计算的状态机**：

- 浅蓝：在深圳
- 中蓝：在香港
- 深蓝：交流日（带星号）
- 黄色：过关日（城市切换日）

系统需要支持：

1. 按模板自动生成色块序列（先落地城市 + 停留天数 + 过关日 + 交流日 + 返程）。
2. 人工编辑单日状态（例如单独更改交流日）。
3. 整段拖拽平移（整体提前/延后），并自动重算资源需求。

### 1.3 标星交流日（深蓝）

- 交流日是高优先级约束节点。
- 需与“姐妹学校可交流日历”做匹配，系统应在冲突时预警：
  - 目标学校不可接待
  - 接待能力不足
  - 同日资源（车、人手、餐厅）冲突

### 1.4 真实工作流在系统中的映射

当前线下流程可抽象为 4 步，HKEIU 应一一支持：

1. **先定模板行程**（小学/中学可不同；先深后港/先港后深可不同）。
2. **按公司承载能力分批**（交通、人员、酒店承压上限）。
3. **生成总览排程**（看每日抵离与住宿总量）。
4. **动态调整**（拖拽平移、改单日交流、冲突修复并广播团队）。

## 2. HKEIU 的核心功能模块（建议的产品边界）

### 2.1 排程总览看板（核心页面）

必须具备：

- 左侧固定信息列：航班号、类型、人数、合计人数、起始城市、批次、学校分组。
- 右侧时间栅格：按天显示团组色块，支持周末/节假日样式。
- 支持“批次折叠/展开”：
  - 折叠看批次级别容量
  - 展开看团组级细节
- 拖拽交互：
  - 拖动整个团组行程（整体平移）
  - 拖动单个节点（例如交流日）
- 实时校验条（冲突提示）：
  - 红色：硬冲突（不可保存）
  - 橙色：软冲突（可保存但需确认）

### 2.2 行程模板引擎

用于把“经验型排程”固化为可复用模板：

- 模板维度：小学/中学/自定义。
- 参数：先落地城市、深港停留天数、第几天过关、第几天交流、总天数。
- 作用：新建团组时一键套用，减少人工涂色。

### 2.3 资源管理中心

统一管理资源主数据及可用性：

- 人员：领队、司机、安全员、协调员（可指派、可请假、可排班）。
- 景点：开放时间、闭馆日、最大接待量、地理位置。
- 姐妹学校：地址、可交流日期、单日接待上限、特殊限制。
- 餐厅：容量、营业时段、适配团餐能力。
- 车辆：座位数、可服务区域、可用时段。

### 2.4 自动统计中心（替代人工汇总）

按日自动输出：

1. **航班需求统计**：当日抵达/离开架次及对应团组。
2. **住宿需求统计**：深圳/香港当晚入住人数（学生+老师）。
   - 关键规则：过关日住宿归属目的地城市。
3. **资源负载统计**：车辆、人手、餐厅、景点是否超载。

### 2.5 协同与通知

- 变更广播：谁在何时把哪个团组改到哪一天。
- 订阅提醒：
  - 交流日变动提醒学校对接人
  - 住宿峰值预警提醒后勤
  - 航班变更提醒交通组
- 审批流（可选）：重要变更需主管审批。

## 3. 数据模型设计（落地实现的最小可行结构）

> 以下是面向实现的关键实体（建议先做 MVP 字段，后续再扩展）。

### 3.1 基础实体

- `School`（学校）
  - 名称、地区、学段、小学/中学标签、联系人。
- `SisterSchoolAvailability`（姐妹学校可交流日）
  - 学校ID、可交流日期、可接待人数上限、备注。
- `Group`（团组）
  - 团组编号、类型、起始城市、学生人数、教师人数、学校列表。
- `Batch`（批次）
  - 批次编号、包含团组、计划抵离日期、共享交通策略。

### 3.2 行程与节点

- `ItineraryPlan`（团组行程主表）
  - 团组ID、开始日期、结束日期、模板来源、当前版本。
- `ItineraryDay`（团组日粒度状态）
  - 日期、城市、日类型（在深/在港/过关/交流/返程）、星标、备注。
- `Milestone`（里程碑）
  - 交流日、过关日、到港日、离港日等关键节点。

### 3.3 运输与住宿

- `FlightSegment`（航段）
  - 团组ID、航班号、起降时间、出发/到达城市、座位需求。
- `BusAssignment`（车辆安排）
  - 批次/团组ID、日期、车辆ID、司机ID、路线。
- `HotelDemandDaily`（每日住宿需求快照）
  - 日期、城市、学生数、教师数、合计人数、来源版本。

### 3.4 资源与冲突

- `ResourceCapacity`（资源容量）
  - 资源ID、日期、总容量、已占用量。
- `ConflictEvent`（冲突事件）
  - 冲突类型、严重等级、影响对象、建议修复动作、状态。

### 3.5 协同审计

- `ChangeLog`（变更日志）
  - 操作人、操作时间、字段前后值、影响范围。
- `Notification`（通知）
  - 消息类型、接收人、是否已读、关联冲突/变更ID。

## 4. 核心计算规则（系统自动化的关键）

### 4.1 合计人数

- `合计人数 = 学生人数 + 教师人数`
- 所有资源需求按合计人数参与计算，必要时可拆分学生/教师结构。

### 4.2 每日住宿人数计算

对每个团组按日期扫描：

1. 若当日状态为“在深圳/交流(深圳)”→ 深圳住宿 + 团组人数。
2. 若当日状态为“在香港/交流(香港)”→ 香港住宿 + 团组人数。
3. 若为过关日 → **目的地城市**住宿 + 团组人数。
4. 若当日离开且不留宿 → 不计入当晚住宿。

再按全团组求和，得到当日两地住宿总数。

### 4.3 每日航班需求统计

- 以航段起降日期为统计口径：
  - 抵达航班（到达深/港）
  - 离开航班（离开深/港）
- 支持展示两种视图：
  - 架次统计（几班）
  - 团组明细（哪些团组）

### 4.4 冲突检测（建议至少实现 8 类）

1. 交流日与姐妹学校不可交流日冲突。
2. 交流日接待人数超上限。
3. 车辆座位不足。
4. 司机或安全员重复占用。
5. 餐厅容量不足。
6. 景点闭馆或超额。
7. 批次抵离窗口与定义不一致。
8. 团组行程断裂（例如没有过关日却城市切换）。

## 5. 页面与交互建议（保证“像线下排程一样快”）

### 5.1 总览页信息架构

- 顶部：日期轴 + 筛选器（小学/中学/批次/城市/冲突等级）。
- 中部：色块甘特网格（团组行）。
- 右侧抽屉：选中团组的详细日程与资源占用。
- 底部：
  - 每日航班需求统计表
  - 每日住宿需求统计表

### 5.2 关键交互

- 拖拽平移后立即显示“变化影响”：
  - 哪天住宿峰值变化
  - 哪些资源从可用变冲突
- 单击星标可切换交流日，并弹出“可交流学校推荐日期”。
- 支持“批次整体移动 + 保持团组相对间距”。

### 5.3 权限与协作

- 角色：排程员、资源管理员、审核主管、查看者。
- 粒度：可按模块/批次授权。
- 审计：任何关键字段变更可追踪并可回滚版本。

## 6. 建议的实施路线（MVP → 完整版）

### 阶段 1（MVP，先把核心跑通）

- 团组/批次管理
- 色块排程编辑（含拖拽平移）
- 每日航班与住宿自动统计
- 基础冲突检测（交流日 + 住宿容量 + 航班窗口）
- 变更通知与日志

### 阶段 2（运营增强）

- 资源中心（人员/车辆/餐厅/景点）
- 更完整冲突引擎
- 批次级模板与智能排期建议
- 审批流与版本对比

### 阶段 3（智能化）

- 自动排程建议（按可交流日、容量、成本优化）
- 峰值风险预测（酒店/车辆/人力）
- 看板大屏与移动端消息联动

## 7. 这套业务在 HKEIU 的“可执行定义”

为了确保系统真正贴合贵中心工作流，可以将“排程成功”定义为：

1. 任一团组可在 3 分钟内完成从模板创建到生成首版日程。
2. 任一变更（交流日调整/整体平移）可在 30 秒内完成并看到冲突与统计重算。
3. 任一日期可即时查看：
   - 抵离航班需求
   - 深/港住宿总人数
   - 关键资源是否超载
4. 任一关键决策（改期、换学校、换资源）都有可追溯记录与通知闭环。

---

## 8. 给管理层的价值总结

HKEIU 不是把 Excel 搬到线上，而是把“经验排程”升级为“可协作、可计算、可预警”的运营系统：

- 从人工汇总变为实时统计
- 从事后发现冲突变为事前预警
- 从个人经验驱动变为团队共享与可复制流程

这将直接提升：

- 排程准确率
- 变更响应速度
- 团队协同效率
- 跨区域交流项目承载能力
